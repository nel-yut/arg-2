# ARG 実装設計書 v4

## 1. 目的
本設計書は、スプレッドシートで管理している ARG ページ群を Web 実装するための仕様を整理するものである。対象はブログ導入、Phase1〜Phase4、Ending を含む全ページであり、検索導線を中心とした進行管理を前提とする。

---

## 2. 技術前提
- フロントエンドは **Vite + React** を第一候補とする。
- 実装初期ではクライアントサイドのみで成立する構成を想定する。
- データソースはスプレッドシートを元にした JSON を利用する。
- JSON 生成スクリプトは後続フェーズで実装する。

---

## 3. URL 設計
### 3.1 URL 形式
各ページは以下の形式で公開する。

```txt
/page-name-00000
```

- `page-name` は英字スラッグ。
- `00000` はスプレッドシート A 列に付与する **5桁の数字文字列**。
- この 5 桁 ID は URL 推測を困難にするために用いる。

### 3.2 スラッグ方針
スラッグは実装時にこちらで命名する。基本ルールは以下。
- 英小文字とハイフンのみ
- 内容を過不足なく表す
- フェーズ番号はスラッグに含めなくてよい

例:
- `blog-entry-01428`
- `dormitory-maintenance-report-18420`
- `tono-saki-record-74012`

### 3.3 URL 直打ち
- **基本は直打ち禁止の設計思想** とする。
- ただし、現時点では 5 桁 ID による推測困難化を優先し、追加のルーティング制御は必須としない。
- 後から必要になれば、未解放ページに対するクライアント側ガードを追加可能とする。

### 3.4 ドメイン分離
- ブログ導入ページ群（Phase0 / Phase-End のブログ系ページ）は **`ino-blog.nel-app`** で公開する。
- 本体サイトは別ドメインで公開する想定とし、設計上は分離された 2 系統のサイトとして扱う。
- ブログから本体サイトへの遷移は、記事本文内の **直接リンクのみ** を用いる。

## 4. フェーズ構成
本作では以下の区分で管理する。

- **Phase0**: ブログ導入
- **Phase1**: 公開サイト探索 → 資料室発見
- **Phase2**: 資料室内調査 → 第3教育寮・反応矯正教育の不穏感
- **Phase3**: 反応矯正教育の深掘り → 遠野紗季 / 榊原澄 / 渡航巡礼へ接続
- **Phase4**: 渡航巡礼の解体 → 死亡根拠の確定
- **Phase5**: Ending
- **Phase-End**: ブログ404 / 終端演出用ページ群

### 4.1 画面上の表示
**すべてのページの右下に小さく Phase とページ番号を表示する。**

例:
- `Phase1 (3)`
- `Phase3 (18)`
- `Phase0 (2)`
- `Phase-End (1)`

用途:
- 制作者・テスターの確認性を上げる
- プレイヤーには露骨すぎない小ささで表示する

## 5. 状態管理
状態は `localStorage` で管理する。

### 5.1 保持項目
最低限保持する項目:

```ts
currentPhase: number
readPages: string[]
```

### 5.2 各項目の意味
- `currentPhase`
  - 現在解放されているフェーズ
  - 初期値は **0** とする
- `readPages`
  - 既読ページの URL または内部 ID の配列
  - ユーザーには現時点では表示しない

### 5.3 フェーズ進行ルール
各 Phase の `(1)` を読むことで、その Phase の `(2)` 以降が検索対象に入る。

より正確には、検索対象は常に以下とする。
- **現在の Phase のページ全体**
- **過去 Phase のページ全体**
- **次 Phase の `(1)` のみ**

例:
- `currentPhase = 1`
  - Phase0 全体
  - Phase1 全体
  - Phase2(1) のみ検索可能
- `currentPhase = 3`
  - Phase0〜Phase3 全体
  - Phase4(1) のみ検索可能

### 5.4 フェーズ更新タイミング
- Phase の `(1)` を閲覧した時点で、その Phase を解放済みとみなす
- あるいは、前 Phase の終端ページで明示的に `currentPhase` を進めてもよい
- 実装時は **ページ閲覧時更新** と **明示イベント更新** のどちらが管理しやすいか比較して決定する

推奨:
- **対象 Phase の `(1)` ページ閲覧時に `currentPhase` を更新**

## 6. 検索仕様
### 6.1 基本仕様
- 検索は **完全一致検索のみ** とする
- 検索対象は「このページを検索するためのワード」列の語のみ
- 部分一致、前方一致、曖昧検索、複数語検索は不可
- 空白を含む検索は不可

### 6.2 検索結果
- **複数件返却** を許可する
- 並び順は **Phase順 → (n)順** とする
- 検索結果には **タイトルのみ** を表示する

### 6.3 検索UI配置
検索UIは以下を除くすべてのページ右上に表示する。

**非表示ページ**
- ブログ
- Phase4(31)

それ以外は共通コンポーネントとして右上固定で表示する。

### 6.4 検索失敗時メッセージ
該当結果がない場合は、通常の 0 件表示ではなく以下の注意文を表示する。

- **「複数ワード検索はできない」**
- **「人名は空白を入れず検索する」**

表示例:

```txt
検索結果が見つかりません。
複数ワード検索はできません。
人名は空白を入れず検索してください。
```

## 7. Excel / JSON データ構造
### 7.1 スプレッドシート想定列
実装時に最低限必要な列は以下。

- A: 5桁ID
- B: Phase
- C: ページ番号
- D: タイトル
- E: 本文
- F: このページを検索するためのワード
- G: コメント
- H: コメント2
- I 以降: 必要に応じて追加

### 7.2 JSON の想定形
```json
{
  "id": "18420",
  "phase": "Phase2",
  "phaseIndex": 7,
  "slug": "third-dormitory-interview-excerpt",
  "path": "/third-dormitory-interview-excerpt-18420",
  "title": "第3教育寮面談記録抜粋",
  "body": "...",
  "searchWords": ["第3教育寮面談記録抜粋"],
  "imagePrompts": ["...", "..."],
  "isSearchVisible": true
}
```

### 7.3 注意
- 検索ワードは **完全一致比較** 用の配列として持つ
- 既存ルール上、ワードは空白なし 1 語を前提とする
- コメント列の画像プロンプトは JSON では `imagePrompts` に変換する

## 8. ページ表示仕様
### 8.1 共通レイアウト
各ページは原則として以下を持つ。
- タイトル
- 本文
- 必要画像
- 右上の検索UI（ブログと Phase4(31) を除く）
- 右下の小さな Phase / ページ番号表示（Phase4(31) も表示する）

### 8.2 画像
- コメント列のプロンプトを元にページ画像を生成 / 配置する
- 1ページにつき最大 2 枚まで
- 2枚必要な場合は `G列`, `H列` を使う
- 画像はヒント埋め込み用にも使用可能

### 8.3 ブログと特殊ページ
- ブログページは通常ページと完全に別レイアウトとする
- ブログページ群は **`ino-blog.nel-app`** で運用する
- ブログから本体サイトへの遷移は **直接リンクのみ** とする
- 本体サイト側からブログへ戻すリンクは必須としない
- Phase-End ページや 404 風ページは専用スタイルでもよい

---

## 9. 導線制御
### 9.1 基本方針
本作はリンク遷移より検索遷移を主とする。よって各本文には、次に検索すべきワードを必ず本文またはタイトルに埋め込む。

### 9.2 実装側で必要なこと
- 各ページに対して「検索ワード」をそのまま検索インデックス化する
- 現在の `currentPhase` に応じて、検索対象ページ集合をフィルタリングする
- 複数件ヒット時は Phase順 → (n)順で表示する

### 9.3 実装側で不要なこと
- 複雑な自然言語検索
- 曖昧検索補正
- 自動サジェスト
- 類義語検索

## 12. 現時点の確定事項まとめ
- Vite + React 前提
- URL は `/page-name-00000`
- 5桁IDは A列で管理
- ブログは **Phase0**
- Ending は **Phase5**
- **Phase-End** も別管理
- `currentPhase` と `readPages` を LocalStorage に保持
- 検索対象は **過去Phase + 現在Phase + 次Phase(1)**
- 検索は **完全一致のみ**
- 結果は **複数件返却 / タイトルのみ / Phase順 → (n)順**
- 検索UIは **ブログ / Phase4(31) 以外の右上**
- Phase4(31) は検索UIを持たないが、右下の Phase 表示は残す
- 0件時は「複数ワード検索不可」「人名は空白なし」を表示
- 既読は保持するが表示しない
- すべてのページ右下に **Phase / ページ番号** を小さく表示
- Phase4(31) も右下の Phase 表示は残す
- ブログは完全別レイアウト、かつ別ドメイン想定
